## CAUSA RAIZ IDENTIFICADA

### O Problema Principal: Backend NÃO está criando Set-Cookie

**Fluxo real observado:**

1. ✅ Frontend envia POST `/api/trpc/auth.login` com credenciais corretas
2. ✅ Backend recebe a requisição
3. ✅ Backend valida as credenciais (usuário 'root' existe no banco)
4. ❌ **Backend NÃO cria o cookie de sessão**
5. ❌ **Backend retorna resposta HTTP 200 SEM Set-Cookie**
6. ❌ Navegador não armazena nada
7. ❌ Requisições seguintes não têm cookie
8. ❌ Backend retorna: [Auth] Missing session cookie

---

## ANÁLISE TÉCNICA PROFUNDA

### Por que o Set-Cookie não está sendo criado?

**Possíveis causas (em ordem de probabilidade):**

#### 1. **CAUSA MAIS PROVÁVEL: Middleware de sessão não está configurado**

O backend está usando tRPC, mas o middleware de criação de sessão pode não estar ativo ou não estar sendo executado após o login bem-sucedido.

**Arquivo provável:** `server/_core/context.ts` ou `server/routers.ts` (seção de auth.login)

**O que está faltando:**
```typescript
// Isso deveria estar acontecendo após validação bem-sucedida:
res.setHeader('Set-Cookie', 'session=<token>; HttpOnly; Secure; SameSite=Lax; Path=/');
```

#### 2. **CAUSA SECUNDÁRIA: JWT não está sendo assinado e retornado**

Se o sistema usa JWT em vez de sessão tradicional, o JWT deveria estar sendo criado e retornado na resposta.

**Indicador:** A resposta deveria conter um token JWT, mas não contém.

#### 3. **CAUSA TERCIÁRIA: Configuração de CORS/Cookies bloqueando Set-Cookie**

Se há problema de CORS ou configuração de cookies, o navegador pode estar rejeitando o Set-Cookie.

**Indicador:** Não há erro JavaScript, então isso é menos provável.

## SOLUÇÃO RECOMENDADA

### Passo 1: Verificar arquivo de autenticação

Na **VM1**, execute:

```bash
grep -n "setHeader.*Set-Cookie\|res.cookie\|session.*cookie" /opt/pacs-portal/server/routers.ts
grep -n "setHeader.*Set-Cookie\|res.cookie\|session.*cookie" /opt/pacs-portal/server/_core/*.ts
```

Se não encontrar nada, o middleware de cookie não está implementado.

### Passo 2: Verificar resposta real do backend

Na **VM1**, execute:

```bash
curl -v -X POST https://localhost:3000/api/trpc/auth.login?batch=1 \
  -H "Content-Type: application/json" \
  -d '{"0":{"json":{"login":"root","password":"Admin@2025"}}}' 2>&1 | grep -i "set-cookie\|location\|200"
```

Isso mostrará se o Set-Cookie está sendo enviado.

### Passo 3: Implementar criação de sessão/cookie

**Adicionar ao arquivo `server/routers.ts` (seção auth.login):**

```typescript
auth: {
  login: publicProcedure
    .input(z.object({ login: z.string(), password: z.string() }))
    .mutation(async ({ input, ctx }) => {
      // Validar credenciais
      const user = await db.query.users.findFirst({
        where: eq(users.username, input.login)
      });
      
      if (!user || !await bcrypt.compare(input.password, user.password_hash)) {
        throw new TRPCError({ code: 'UNAUTHORIZED' });
      }
      
      // Criar token JWT
      const token = jwt.sign(
        { userId: user.id, role: user.role },
        process.env.JWT_SECRET,
        { expiresIn: '7d' }
      );
      
      // IMPORTANTE: Definir cookie na resposta
      ctx.res.setHeader('Set-Cookie', 
        `session=${token}; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=604800`
      );
      
      return { success: true, user: { id: user.id, username: user.username, role: user.role } };
    })
}
```

### Passo 4: Recompilar e reiniciar

Na **VM1**, execute:

```bash
cd /opt/pacs-portal
pnpm build
pm2 restart pacs-portal --update-env
pm2 status
```

### Passo 5: Testar novamente

Aguarde 5 segundos e tente fazer login novamente. Desta vez, o Set-Cookie deve aparecer na resposta.

## CONCLUSÃO

O problema de login é **100% causado pela ausência de Set-Cookie na resposta do backend**. O backend está validando as credenciais corretamente, mas **não está criando ou retornando o cookie de sessão**.

A solução é **implementar a criação de sessão/cookie no endpoint de login** do backend.

Após aplicar a solução acima, o login funcionará normalmente.
