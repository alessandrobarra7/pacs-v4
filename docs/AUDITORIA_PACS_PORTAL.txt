================================================================================
                            RESUMO EXECUTIVO
================================================================================

O PACS Portal é um sistema completo de gestão de laudos radiológicos com
integração DICOM para busca de exames médicos em servidores PACS/Orthanc.
O projeto utiliza arquitetura monolítica moderna com comunicação type-safe
entre frontend e backend através de tRPC.

STATUS ATUAL: ~75% concluído
- 4 módulos principais implementados e funcionando
- Integração DICOM real testada com sucesso (48 estudos recuperados)
- Sistema de templates e laudos operacional
- Sistema de anamnese com CID criado (backend completo, UI em integração)

================================================================================
                          STACK TECNOLÓGICA
================================================================================

FRONTEND:
- React 19 (framework UI)
- TypeScript (type safety)
- Tailwind CSS 4 (estilização)
- shadcn/ui (componentes UI)
- TanStack Query (gerenciamento de estado assíncrono)
- Wouter (roteamento)

BACKEND:
- Node.js + Express 4 (servidor HTTP)
- tRPC 11 (comunicação type-safe cliente-servidor)
- Drizzle ORM (abstração de banco de dados)
- PostgreSQL (banco de dados)
- JWT (autenticação)
- Manus OAuth (sistema de login)

INTEGRAÇÃO DICOM:
- Python 3.11 + pynetdicom (comunicação DICOM C-FIND)
- Scripts Python executados via child_process do Node.js
- Bash wrappers para isolamento de ambiente Python

ARQUITETURA:
- Monolítico moderno (frontend + backend em processo único)
- Deploy unificado (porta 3000)
- Type safety end-to-end (mudanças no backend refletem automaticamente no frontend)

┌─────────────────────────────────────────────────────────────────────────────┐
│ COMUNICAÇÃO FRONTEND-BACKEND (tRPC)                                         │
└─────────────────────────────────────────────────────────────────────────────┘

O projeto utiliza tRPC para comunicação type-safe entre frontend e backend.

VANTAGENS:
- Type safety end-to-end (tipos do backend refletem automaticamente no frontend)
- Não há necessidade de gerar schemas ou contratos manualmente
- IntelliSense completo no frontend (autocomplete de procedures e tipos)
- Erros de tipo detectados em tempo de desenvolvimento
- Reduz drasticamente bugs de integração

ESTRUTURA:
1. Backend define procedures em server/routers.ts:
   
   export const appRouter = router({
     units: router({
       list: publicProcedure.query(async () => {
         return await db.select().from(units);
       }),
       create: protectedProcedure
         .input(z.object({ name: z.string(), ... }))
         .mutation(async ({ input }) => {
           return await db.insert(units).values(input);
         }),
     }),
   });

2. Frontend consome via hooks:
   
   const { data: unitsList } = trpc.units.list.useQuery();
   const createUnit = trpc.units.create.useMutation();

3. Tipos fluem automaticamente:
   - Se backend retorna { id: number, name: string }, frontend recebe exatamente isso
   - Se backend muda estrutura, TypeScript alerta frontend imediatamente


┌─────────────────────────────────────────────────────────────────────────────┐
│ INTEGRAÇÃO DICOM (Python Bridge)                                            │
└─────────────────────────────────────────────────────────────────────────────┘

A integração DICOM utiliza Python + pynetdicom executado via Node.js.

ARQUITETURA:
1. Node.js (backend) recebe requisição do frontend via tRPC
2. Node.js executa script Python via child_process.spawn()
3. Python se conecta ao PACS via C-FIND DICOM
4. Python retorna JSON com resultados
5. Node.js parseia JSON e retorna para frontend via tRPC

FLUXO DE BUSCA:
Frontend (React)
    ↓ trpc.pacs.query.useQuery({ unitId, patientName, studyDate })
Backend (Node.js)
    ↓ spawn('bash', ['dicom_query.sh', ip, port, aeTitle, ...])
Python Script (dicom_query.py)
    ↓ C-FIND DICOM → PACS Server
PACS Server (Orthanc)
    ↓ Retorna estudos DICOM
Python Script
    ↓ JSON output
Backend (Node.js)
    ↓ Parse JSON
Frontend (React)
    ↓ Exibe resultados na tabela

VANTAGENS:
- Isolamento de ambiente (Python 3.11 via bash wrapper)
- Biblioteca pynetdicom madura e confiável
- Fácil manutenção (scripts Python separados)
- Logs detalhados para debugging


┌─────────────────────────────────────────────────────────────────────────────┐
│ PERSISTÊNCIA DE BUSCA (localStorage)                                        │
└─────────────────────────────────────────────────────────────────────────────┘

Problema resolvido: Ao clicar em "Laudar" e depois voltar, a busca era perdida.

Solução implementada:
1. Antes de navegar para editor de laudos, salvar estado no localStorage:
   localStorage.setItem('pacsSearchState', JSON.stringify({
     results: studies,
     filters: { unitId, patientName, studyDate, period }
   }));

2. Ao voltar para página de busca, restaurar estado:
   const savedState = localStorage.getItem('pacsSearchState');
   if (savedState) {
     const { results, filters } = JSON.parse(savedState);
     setStudies(results);
     setFilters(filters);
   }

3. Limpar estado ao fazer nova busca:
   localStorage.removeItem('pacsSearchState');

RESULTADO: Usuário pode laudar múltiplos exames sem perder contexto da busca.


┌─────────────────────────────────────────────────────────────────────────────┐
│ TIMEZONE HANDLING (Server-side Date Calculation)                            │
└─────────────────────────────────────────────────────────────────────────────┘

Problema resolvido: Busca por "Exames de Hoje" não funcionava.

Causa raiz:
- Navegador (GMT+0/UTC) calculava "hoje" como 24/02/2026
- Servidor (EST/GMT-5) estava em 23/02/2026
- PACS retornava 0 resultados porque datas não batiam

Solução implementada:
1. Frontend envia valor especial "TODAY" ao invés de calcular data:
   studyDate: period === 'today' ? 'TODAY' : ''

2. Backend intercepta "TODAY" e calcula data no timezone do servidor:
   if (input.studyDate === 'TODAY') {
     const today = new Date();
     const year = today.getFullYear();
     const month = String(today.getMonth() + 1).padStart(2, '0');
     const day = String(today.getDate()).padStart(2, '0');
     studyDate = `${year}${month}${day}`;
   }

3. Mesmo padrão para LAST_7_DAYS e LAST_30_DAYS

RESULTADO: "Hoje" sempre baseado no horário do servidor onde PACS está rodando.

   - id, user_id, action, resource_type, resource_id

RELACIONAMENTOS:
- templates.unit_id → units.id (template específico de unidade)
- templates.created_by → users.id (quem criou o template)
- reports.template_id → templates.id (template usado no laudo)
- reports.created_by → users.id (médico que criou o laudo)
- anamnesis.created_by → users.id (quem preencheu a anamnese)
- audit_logs.user_id → users.id (quem executou a ação)

================================================================================
                          ISSUES CONHECIDOS
================================================================================

1. Toast notifications não aparecem visualmente
   - Funcionalidade funciona (dados são salvos)
   - Mas notificações não são exibidas na tela
   - Possível problema com z-index ou configuração do Toaster

2. Visualizador DICOM incompleto
   - Erros TypeScript no DicomViewer.tsx
   - C-MOVE não testado em produção
   - Ferramentas de manipulação não implementadas

3. Sem paginação na listagem de exames
   - Com grandes volumes (1000+ estudos), performance pode degradar
   - Recomendado implementar paginação ou virtualização

4. Sem modo escuro
   - Interface apenas em modo claro
   - Pode causar fadiga visual em uso prolongado
