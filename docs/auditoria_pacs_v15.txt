── V2 ── GRAVIDADE: ALTA — PERFORMANCE
PROBLEMA: getSystemOwnerLiveByUnit executa 4 queries sequenciais por unidade
          dentro de um for-loop (N+1 clássico)

LOCALIZAÇÃO: server/db.ts, função getSystemOwnerLiveByUnit, linha 2783

Para cada unidade ativa, dentro do loop:
  1. SELECT responsável financeiro ativo            (1 query)
  2. SELECT preço do sistema ativo                  (1 query)
  3. SELECT ciclo aberto do sistema + summary        (1 query)
  4. SELECT custo médico (billing_cycle_doctor_summary) (1 query)

Com 10 unidades: 1 query inicial + 40 queries = 41 queries por chamada.

Resultado: a cada minuto, com 20 unidades, o servidor executa 81 queries
para servir esse endpoint. Em um banco MySQL remoto com 5ms de latência
por query, isso significa ~400ms de espera mínima por requisição.

── V10 ── GRAVIDADE: BAIXA
PROBLEMA: SLA com source='direct_sign' registra became_ready_at = momento
          da assinatura, não o momento real de chegada do exame

LOCALIZAÇÃO: server/routers/sla.ts, função ensureReadinessExists, linha 190
  became_ready_at: now,  // "now" = momento do sign, não da chegada do exame

Quando um laudo é assinado sem que tenha passado por anamnesisSimple,
o readiness é criado com became_ready_at = hora da assinatura. O campo
delay_seconds é calculado como (reportedAt - dueAt), e dueAt = became_ready_at
+ SLA. Portanto, para todo exame "direct_sign", delay_seconds será sempre
um número muito pequeno (tempo entre criar e fechar o readiness, milissegundos).

Relatórios de SLA mostrarão conformidade artificial de 100% para esses
exames. A métrica fica distorcida.

Isso é uma limitação de design: não há forma de saber retroativamente
quando o exame "ficou pronto para laudar" sem a anamnese. A solução
correta é registrar o readiness quando o exame entra no studies_cache
(chegada real), não na assinatura.

CORREÇÃO (ver Parte 5, item C10).

PARTE 4 — OPINIÕES E RECOMENDAÇÕES DE MELHORIA (visão global)
================================================================================

Esta seção vai além dos bugs pontuais. Trata de decisões de arquitetura,
organização de código e qualidade que afetam o sistema como um todo.

────────────────────────────────────────────────────────────────────────────────
4.1  DOIS SISTEMAS DE BILLING COEXISTINDO: DECIDIR E MIGRAR
────────────────────────────────────────────────────────────────────────────────

O maior problema estrutural do sistema financeiro continua sendo a
coexistência de dois subsistemas de billing:

  Sistema A: billing_report_items + billing_monthly_*
    → Populado via calculateCompetence (batch manual)
    → Usado por getDoctorSummary (aba "extrato" do médico)
    → Usado pelo FinanceDashboard (totalSystem, totalDoctors do mês)

  Sistema B: billing_visit_events + billing_cycles + billing_cycle_*_summary
    → Populado em tempo real no sign do laudo
    → Usado por getDoctorProduction, getSystemOwnerLiveByUnit, tudo novo

Toda funcionalidade nova está corretamente no Sistema B.
O Sistema A só permanece para não quebrar funcionalidades existentes.

RECOMENDAÇÃO: Definir formalmente o Sistema A como "arquivo histórico
de períodos fechados" e o Sistema B como "fonte operacional corrente".
Implementar um job de fechamento que:
  1. Ao fechar um ciclo, snapshot o Sistema B para o Sistema A
  2. Remova as chamadas ao Sistema A das telas de saldo do dia a dia

Enquanto isso não acontece, o médico continuará vendo saldos diferentes
dependendo de qual aba clica.

────────────────────────────────────────────────────────────────────────────────
4.2  PADRÃO N+1 PRECISA SER ELIMINADO SISTEMATICAMENTE
────────────────────────────────────────────────────────────────────────────────

Foram identificados pelo menos dois casos de N+1 nessa versão (V2 e V3).
Provavelmente existem outros em funções não auditadas.

O padrão aparece quando o código faz uma query inicial para buscar uma
lista, depois itera sobre cada item fazendo novas queries para enriquecer
os dados.

RECOMENDAÇÃO: Usar JOINs ou, quando JOINs não são viáveis (ex: queries
para tabelas diferentes), usar `inArray` com um único SELECT:

  // Ruim (N+1):
  for (const row of rows) {
    const extra = await db.select().from(t).where(eq(t.id, row.related_id));
  }

  // Bom (1 query com JOIN):
  const rows = await db.select({ main: t1, extra: t2 })
    .from(t1).leftJoin(t2, eq(t1.related_id, t2.id));

  // Bom (2 queries planas):
  const ids = rows.map(r => r.related_id).filter(Boolean);
  const extras = await db.select().from(t2).where(inArray(t2.id, ids));
  const extrasMap = new Map(extras.map(e => [e.id, e]));

────────────────────────────────────────────────────────────────────────────────
4.3  PAGINAÇÃO É AUSENTE EM TODOS OS ENDPOINTS DE LISTAGEM PESADA
────────────────────────────────────────────────────────────────────────────────

getDoctorStatement: limit(500)
getResponsibleDebtByDoctor: limit(1000)
listSystemReceivables: sem limit explícito (pode retornar todos os ciclos)
listBillingReportItems: sem limit

Para sistemas médicos que acumulam dezenas de laudos por dia por unidade,
um único médico pode ultrapassar 500 eventos por mês em 2-3 meses de uso.
A partir desse ponto, os extratos ficam incompletos sem qualquer aviso.

RECOMENDAÇÃO: Implementar cursor-based pagination ou offset pagination:

  // Input com paginação:
  .input(z.object({
    page: z.number().int().min(1).default(1),
    pageSize: z.number().int().min(10).max(200).default(50),
    // ... outros filtros
  }))

  // Query com offset:
  .limit(input.pageSize)
  .offset((input.page - 1) * input.pageSize)

  // Retorno com metadata:
  return { data: rows, total, page, pageSize, hasMore: total > page * pageSize };

────────────────────────────────────────────────────────────────────────────────
4.4  TIPAGEM `any` ESPALHADA — RISCO DE BUG SILENCIOSO
────────────────────────────────────────────────────────────────────────────────

Em múltiplos places do código:
  const conditions: any[] = []
  const updateData: Record<string, any> = {}
  (result[0] as any).insertId

O uso de `any` derrota o TypeScript. Um campo com nome errado no
`updateData` não gera erro em compilação — só falha silenciosamente
em runtime (query não afeta nenhum campo sem erro).

RECOMENDAÇÃO: Tipar explicitamente os arrays de condições e os objetos
de update usando os tipos do Drizzle:

  import { SQL } from 'drizzle-orm';
  const conditions: SQL[] = [];

  // Para updates parciais, use Partial do tipo da tabela:
  type UpdateData = Partial<typeof unit_doctor_compensation_rules.$inferInsert>;
  const updateData: UpdateData = {};

────────────────────────────────────────────────────────────────────────────────
4.5  IMPORTS DINÂMICOS EM RUNTIME SÃO DESNECESSÁRIOS E LENTOS
────────────────────────────────────────────────────────────────────────────────

Em praticamente todos os procedures do billing.ts:
  const { billing_cycles } = await import('../../drizzle/schema');
  const { eq, and } = await import('drizzle-orm');

Esses imports são resolvidos em runtime para cada chamada de procedure.
O módulo já está em memória após o primeiro import, então o impacto é
pequeno — mas o padrão é desnecessário e dificulta a leitura.

RECOMENDAÇÃO: Mover todos os imports para o topo do arquivo:
  import { billing_cycles, billing_visit_events, ... } from '../../drizzle/schema';
  import { eq, and, or, inArray, ... } from 'drizzle-orm';

────────────────────────────────────────────────────────────────────────────────
4.6  FALTA UM SISTEMA DE MIGRAÇÃO DE BANCO RASTREÁVEL
────────────────────────────────────────────────────────────────────────────────

O schema.ts cresceu para 808 linhas com tabelas novas adicionadas
incrementalmente (billing_cycles ganhou paid_status, paid_at, etc. nessa
versão). Não há migrations numeradas e auditáveis para essas mudanças.

A pasta /scripts/ contém scripts de migração ad hoc (.mjs) com nomes
não padronizados, sem garantia de idempotência.

RECOMENDAÇÃO: Usar Drizzle migrations (drizzle-kit generate) para todas
as mudanças de schema daqui em diante. Cada alteração gera um arquivo
numerado em drizzle/migrations/ que pode ser executado de forma controlada
e auditada.

────────────────────────────────────────────────────────────────────────────────
4.7  AUSÊNCIA DE TESTES AUTOMATIZADOS NO MÓDULO FINANCEIRO
────────────────────────────────────────────────────────────────────────────────

O módulo mais crítico do sistema (billing) não tem nenhum teste unitário
ou de integração. As funções de db.ts têm lógica complexa de cálculo
(getEconomicDashboard, getSystemOwnerLiveByUnit, calculateCompetence)
que foi corrigida em múltiplas versões porque só quebrou em produção.

RECOMENDAÇÃO: Escrever ao menos testes para as funções de cálculo:
  - calculateCompetence: verificar que N laudos geram N eventos
  - getEconomicDashboard: verificar que periodicity='quarterly' retorna amount/3
  - getSystemOwnerLiveByUnit: verificar que net_amount = system - doctor
  - resetDoctorBilling: verificar que consolidados são decrementados

────────────────────────────────────────────────────────────────────────────────
4.8  NOMES DE CAMPOS INCONSISTENTES ENTRE TABELAS
────────────────────────────────────────────────────────────────────────────────

billing_cycle_doctor_summary: paid_at / paid_by
billing_cycles: paid_at / paid_by_user_id

billing_visit_events: system_paid_at
billing_cycle_system_summary: paid_at / paid_by

A mesma informação (quem marcou como pago e quando) usa nomes diferentes
em tabelas diferentes. Isso cria confusão e erros quando um desenvolvedor
assume que o campo chama paid_by em todas as tabelas.

RECOMENDAÇÃO: Padronizar para paid_by_user_id em todas as tabelas que
armazenam esse dado, e alinhar na próxima migration.

────────────────────────────────────────────────────────────────────────────────
4.9  COMPONENTE SlaCountdown PODE SER MAIS INFORMATIVO
────────────────────────────────────────────────────────────────────────────────
