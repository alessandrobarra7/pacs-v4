LÓGICA FINAL ESPERADA DO SISTEMA
----------------------------------
1. Cada laudo assinado gera 1 evento financeiro.
2. Cada unidade possui sua própria configuração financeira.
3. Cada médico recebe conforme o valor definido para aquela unidade.
4. Cada responsável financeiro visualiza apenas suas unidades.
5. Todos os totais usam o ciclo financeiro real da unidade.
6. O que aparece na tela é exatamente o que será pago.

PONTO 2 — ALTA PRIORIDADE
setFinancialEnabled sem validação de readiness no backend
================================================================================

CONFIRMADO PELO DOCUMENTO. IDENTIFICADO NO CÓDIGO.

setFinancialEnabled (linha 2864 de financeSimple.ts) executa diretamente:
  await db.update(units).set({ financial_enabled: input.enabled })...

Não chama unitFinancialReadiness. Não verifica se a unidade está pronta.
Um admin pode ativar o financeiro de uma unidade sem ciclo, sem responsável,
sem preços configurados. A proteção existe apenas no frontend (botão desabilitado).

RISCO: contornar o frontend (chamada direta à API) ativa o financeiro
em unidade sem configuração.

CORREÇÃO — Adicionar validação no backend:

  Em setFinancialEnabled, quando input.enabled === true:

  // Não permitir ativação sem readiness completo
  if (input.enabled) {
    const unitRow = await db.select({
      isActive: units.isActive,
      s: units.billing_cycle_start_day,
      e: units.billing_cycle_end_day,
      sys: units.default_system_price,
      doc: units.default_doctor_price,
    }).from(units).where(eq(units.id, input.unit_id)).limit(1);

    const u = unitRow[0];
    if (!u?.isActive) {
      throw new TRPCError({ code: 'BAD_REQUEST',
        message: 'A unidade precisa estar ativa para habilitar o financeiro.' });
    }
    if (!u.s || !u.e) {
      throw new TRPCError({ code: 'BAD_REQUEST',
        message: 'Configure o ciclo financeiro antes de ativar.' });
    }

    const responsible = await getActiveResponsibleForUnit(input.unit_id, new Date());
    if (!responsible) {
      throw new TRPCError({ code: 'BAD_REQUEST',
        message: 'Vincule um responsável financeiro antes de ativar.' });
    }

    const sysPrice = await getActiveSystemPrice(
      responsible.financial_responsible_id, input.unit_id, new Date()
    );
    const hasPrice = sysPrice || (u.sys && Number(u.sys) > 0);
    if (!hasPrice) {
      throw new TRPCError({ code: 'BAD_REQUEST',
        message: 'Configure o preço do sistema antes de ativar.' });
    }
  }

  Registrar ativação/desativação no audit_log:
    await createAuditLog({
      user_id: ctx.user.id,

      action: input.enabled ? 'FINANCIAL_ENABLED' : 'FINANCIAL_DISABLED',

PONTO 4 — ALTA PRIORIDADE
Sem validação de escopo por unidade — responsavel_financeiro pode acessar qualquer unidade
================================================================================

IDENTIFICADO NO CÓDIGO. CITADO PELO DOCUMENTO.

assertAdmin inclui responsavel_financeiro. Procedures como markDoctorPaid,
eventsByDoctorUnit e unitFinancialReadiness recebem qualquer unit_id sem verificar
se aquele responsável financeiro tem vínculo com aquela unidade.

Um responsavel_financeiro pode, via chamada direta à API:
- ver eventos de unidades que não são suas
- marcar pagamentos em unidades que não gerencia

CORREÇÃO — Criar helper assertCanAccessFinancialUnit:

  Adicionar em financeSimple.ts após assertAdmin (linha ~155):

  async function assertCanAccessFinancialUnit(
    db: NonNullable<Awaited<ReturnType<typeof getDb>>>,
    user: { id: number; role: string },
    unitId: number
  ): Promise<void> {
    // admin_master acessa tudo
    if (user.role === 'admin_master') return;

    // unit_admin: verificar vínculo de administração
    if (user.role === 'unit_admin') {
      const perm = await db
        .select({ unit_id: user_unit_permissions.unit_id })
        .from(user_unit_permissions)
        .where(and(
          eq(user_unit_permissions.user_id, user.id),
          eq(user_unit_permissions.unit_id, unitId),
        ))
        .limit(1);
      if (!perm.length) {
        throw new TRPCError({ code: 'FORBIDDEN',
          message: 'Sem acesso a esta unidade.' });
      }
      return;
    }

    // responsavel_financeiro: verificar vínculo financeiro
    if (user.role === 'responsavel_financeiro') {
      const responsibleId = await getResponsibleIdForUser(user.id);
      if (!responsibleId) {
        throw new TRPCError({ code: 'FORBIDDEN',
          message: 'Sem responsável financeiro vinculado.' });
      }
      const link = await db
        .select({ id: financial_responsible_units.id })
        .from(financial_responsible_units)
        .where(and(
          eq(financial_responsible_units.financial_responsible_id, responsibleId),
          eq(financial_responsible_units.unit_id, unitId),
          isNull(financial_responsible_units.ends_at),
        ))
        .limit(1);
      if (!link.length) {
        throw new TRPCError({ code: 'FORBIDDEN',
          message: 'Esta unidade não está vinculada ao seu perfil financeiro.' });
      }
      return;
    }

    throw new TRPCError({ code: 'FORBIDDEN' });
  }

  Aplicar em todas as procedures que recebem unit_id:

  Procedures prioritárias para aplicação imediata:
    markDoctorPaid       → await assertCanAccessFinancialUnit(db, ctx.user, input.unit_id)
    eventsByDoctorUnit   → await assertCanAccessFinancialUnit(db, ctx.user, input.unit_id)
    unitFinancialReadiness → await assertCanAccessFinancialUnit(db, ctx.user, input.unit_id)
    setDoctorPriceDirect → await assertCanAccessFinancialUnit(db, ctx.user, input.unit_id)

    repriceMissingEvents → await assertCanAccessFinancialUnit(db, ctx.user, input.unit_id)

PONTO 8 — MÉDIA PRIORIDADE
Política de laudos retificados e cancelados indefinida
================================================================================

CITADO PELO DOCUMENTO. CONFIRMADO PELA ANÁLISE.

Quando um laudo é retificado (revise), o sistema atualiza apenas
report_status_snapshot para 'revised' em billing_visit_events.
Nenhum ajuste financeiro é gerado. O valor do laudo permanece o mesmo.

O schema de billing_visit_events tem um enum com 'cancelled' em
readiness_status, mas não existe campo de status financeiro do evento
(active / cancelled / reversed / adjusted).

REGRA RECOMENDADA (conforme documento):

  Correção textual do laudo → manter evento, sem ajuste.
  Cancelamento do laudo → marcar evento como cancelled, não apagar.
  Troca de médico responsável → estornar evento antigo, criar novo.
  Troca de unidade → estornar evento antigo, criar novo na unidade correta.

CORREÇÃO — PARTE A: adicionar campo de status ao evento

  Migration:
    ALTER TABLE billing_visit_events
      ADD COLUMN financial_status ENUM('active', 'cancelled', 'reversed', 'adjusted')
        NOT NULL DEFAULT 'active'
        AFTER pricing_status;

  Schema Drizzle:
    financial_status: mysqlEnum("financial_status",
      ["active", "cancelled", "reversed", "adjusted"]
    ).notNull().default("active"),

CORREÇÃO — PARTE B: procedure de cancelamento de evento

  cancelBillingEvent: protectedProcedure
    .input(z.object({
      event_id: z.number().int(),
      reason: z.string().min(5),
    }))
    .mutation(async ({ input, ctx }) => {
      if (ctx.user.role !== 'admin_master')
        throw new TRPCError({ code: 'FORBIDDEN' });
      const db = await getDb();
      if (!db) throw new TRPCError({ code: 'INTERNAL_SERVER_ERROR' });

      await db
        .update(billing_visit_events)
        .set({ financial_status: 'cancelled' })
        .where(eq(billing_visit_events.id, input.event_id));

      await createAuditLog({
        user_id: ctx.user.id,
        action: 'BILLING_EVENT_CANCELLED',
        target_type: 'BILLING_EVENT',
        target_id: String(input.event_id),
      });

      return { ok: true };
    }),

CORREÇÃO — PARTE C: filtrar eventos cancelados das somas

  Em todas as queries de summary, adicionar:
    ne(billing_visit_events.financial_status, 'cancelled')
