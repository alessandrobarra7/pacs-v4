PROBLEMA 1 — MÉDIO · shared/permissions.ts
canAccessUnit duplicado com comportamentos opostos em dois arquivos
================================================================================

CAUSA
  O nome canAccessUnit existe em dois arquivos:

  shared/permissions.ts     → síncrona, recebe PermissionCheck,
                              retorna boolean baseado APENAS em role
                              (não consulta banco)

  server/authorization.ts   → assíncrona, recebe (user, unitId, permission),
                              CONSULTA o banco de user_unit_permissions

  Um import acidental de permissions.ts em vez de authorization.ts
  usaria a versão sem banco, "autorizando" com base apenas em role —
  sem verificar a permissão granular real do usuário na unidade.

ARQUIVO
  shared/permissions.ts

LOCALIZAR (linha ~100):
  export function canAccessUnit(check: PermissionCheck): boolean {
    if (check.role === 'admin_master') {
      return true;
    }
    return check.unitId === check.targetUnitId;
  }

ANTES
  export function canAccessUnit(check: PermissionCheck): boolean {
    if (check.role === 'admin_master') return true;
    return check.unitId === check.targetUnitId;
  }

DEPOIS
  // FIX: renomear para evitar colisão de nome com authorization.canAccessUnit
  // A versão canônica (que consulta banco) está em server/authorization.ts
  // Esta função é apenas para lógica de role puro, sem banco
  export function canAccessUnitByRole(check: PermissionCheck): boolean {
    if (check.role === 'admin_master') return true;
    return check.unitId === check.targetUnitId;
  }

BUSCAR CHAMADAS para atualizar:
  grep -rn "canAccessUnit" client/ server/ shared/ --include="*.ts" --include="*.tsx"

  Para cada ocorrência que importa de permissions.ts:
    import { canAccessUnit } from '../../shared/permissions'    → renomear para canAccessUnitByRole
    import { canAccessUnit } from '@shared/permissions'         → renomear para canAccessUnitByRole

  Ocorrências que importam de authorization.ts permanecem inalteradas.

PROBLEMA 2 — MÉDIO · server/auth.service.ts
Dois sistemas JWT coexistindo — verifySession e JWT_SECRET são código morto
================================================================================

CAUSA
  O procedure login usa sdk.signSession() (Manus SDK) para criar tokens.
  O SDK também verifica os tokens internamente via sdk.verifySession()
  no contexto tRPC. O AuthService.verifySession() usa jwt.verify() com
  um JWT_SECRET próprio — mas NUNCA é chamado em nenhum lugar.
  O JWT_SECRET e o SESSION_DURATION também são código morto.

  Dois sistemas paralelos criam falsa sensação de segurança e confusão
  sobre qual mecanismo está efetivamente protegendo as rotas.

ARQUIVO
  server/auth.service.ts

REMOVER os seguintes blocos inteiros:

  1. A função getJwtSecret() completa:
       function getJwtSecret(): string { ... }

  2. As constantes derivadas:
       const JWT_SECRET = getJwtSecret();
       const SESSION_DURATION = ENV.sessionDurationHours * 60 * 60 * 1000;

  3. A interface SessionPayload (se não usada em outro lugar):
       export interface SessionPayload { ... }
       Verificar antes: grep -rn "SessionPayload" server/ shared/

  4. O método verifySession do AuthService:
       static async verifySession(token: string): Promise<SessionPayload> {
         try {
           const payload = jwt.verify(token, JWT_SECRET) as SessionPayload;
           return payload;
         } catch (error) {
           throw new Error('INVALID_SESSION');
         }
       }

ADICIONAR comentário explicando a arquitetura de sessão:
  // ARQUITETURA DE SESSÃO:
  // Tokens são criados por sdk.signSession() e verificados por sdk.verifySession()
  // O SDK gerencia o ciclo de vida dos tokens (criação, verificação, expiração).
  // A verificação acontece automaticamente no contexto tRPC (_core/context.ts).
  // Não existe JWT_SECRET local — a chave é gerenciada pelo SDK.

TAMBÉM VERIFICAR se jwt pode ser removido das dependências:
  grep -rn "from 'jsonwebtoken'\|require('jsonwebtoken')" server/
  Se só estava sendo usado no auth.service.ts, remover do package.json:
    pnpm remove jsonwebtoken @types/jsonwebtoken

PROBLEMA 4 — BAIXO · server/auth.service.ts
expiration_date comparada em UTC — inconsistente com TZ=America/Fortaleza
================================================================================

CAUSA
  new Date().toISOString().slice(0,10) retorna a data em UTC.
  Com TZ=America/Fortaleza (UTC-3), entre 21h e 00h o "hoje" em UTC
  é o "amanhã" local. Uma conta que expira "hoje" 30/04 poderia ser
  rejeitada às 21h do dia 30/04 porque toISOString() retornaria '2026-05-01'.

ARQUIVO
  server/auth.service.ts

LOCALIZAR (linha ~50):
  const today = new Date().toISOString().slice(0, 10); // 'YYYY-MM-DD'

ANTES
  const today = new Date().toISOString().slice(0, 10); // 'YYYY-MM-DD'

DEPOIS
  // FIX: usar timezone do processo (TZ=America/Fortaleza) em vez de UTC
  // en-CA formata como YYYY-MM-DD nativamente — mesmo padrão do toDicomDateInTimeZone
  const today = new Intl.DateTimeFormat('en-CA', {
    timeZone: process.env.TZ ?? 'America/Fortaleza',
  }).format(new Date());

PROBLEMA 7 — BAIXO · Nginx / server/_core/index.ts
Sem proteção contra brute-force no endpoint de login
================================================================================

CAUSA
  O procedure login não tem limitação de tentativas por IP.
  Com bcrypt custo 12 (~300ms/hash), um atacante com CPU potente pode
  tentar ~3 senhas/segundo por IP. Com múltiplos IPs, o ataque escala.
  Para um sistema médico, rate limiting no login é boa prática de segurança.

OPÇÃO A — Rate limiting via Nginx (sem alterar código Node)
  Mais simples e eficiente. Adicionar ao arquivo de configuração Nginx da VM1:

    # Arquivo: /etc/nginx/conf.d/pacs-portal.conf (ou sites-available/pacs-portal)

    # Zona de rate limiting para login — 10 req/minuto por IP
    limit_req_zone $binary_remote_addr zone=login_zone:10m rate=10r/m;

    # Dentro do bloco server { ... }:
    location /trpc/auth.login {
      limit_req zone=login_zone burst=5 nodelay;
      limit_req_status 429;
      # ... demais configurações de proxy_pass ...
    }

  Aplicar e testar:
    sudo nginx -t
    sudo nginx -s reload

OPÇÃO B — Rate limiting via express-rate-limit no Node (requer package)
  Se preferir controlar no código:

    pnpm add express-rate-limit

  Em server/_core/index.ts, ADICIONAR após os imports:

    import rateLimit from 'express-rate-limit';

    const loginRateLimiter = rateLimit({
      windowMs: 15 * 60 * 1000, // janela de 15 minutos
      max: 10,                   // máximo 10 tentativas por IP
      message: { error: 'Muitas tentativas. Aguarde 15 minutos.' },
      standardHeaders: true,
      legacyHeaders: false,
      skipSuccessfulRequests: true, // não conta logins bem-sucedidos
    });

  APLICAR antes do handler tRPC, apenas para a rota de login:
    app.use('/trpc/auth.login', loginRateLimiter);

  Ou de forma mais abrangente para todos os endpoints públicos:
    app.use('/trpc/auth', loginRateLimiter);

RECOMENDAÇÃO
  Usar Opção A (Nginx) se o servidor já tem Nginx na frente do Node.
  É mais eficiente (o Node nem processa requisições bloqueadas) e
  não requer alterar o código da aplicação.
  Combinar com Opção B para proteção dupla é o ideal.
