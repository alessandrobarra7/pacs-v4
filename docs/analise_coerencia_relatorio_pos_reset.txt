SEÇÃO 7 — "Não existe base financeira mínima recriada"
────────────────────────────────────────────────────────────────────────────────

STATUS: ✅ CORRETO — O diagnóstico está certo. Após o reset, as tabelas estão
        zeradas e precisam ser reconfiguradas manualmente.

Análise:
  O sistema não tem auto-configuração financeira. O fluxo correto é:
  1. Admin_master cria responsável financeiro
  2. Vincula responsável à unidade
  3. Configura preços (sistema e médico)
  4. Configura ciclo financeiro
  Sem essas etapas, billing_visit_events não terão financial_responsible_id
  e os valores serão NULL.

  PORÉM: o código atual tem um FALLBACK importante que o relatório não menciona.
  Em createBillingVisitEvent (db.ts, linha 1800), se não houver responsável
  financeiro configurado, o sistema usa default_system_price e
  default_doctor_price da tabela units como fallback.
  Isso significa que billing_visit_events SERÃO CRIADOS mesmo sem responsável,
  mas com financial_responsible_id = NULL e usando os preços padrão da unidade.

Ação necessária:
  Configurar as unidades com default_system_price e default_doctor_price
  como valores de fallback enquanto o responsável financeiro não é recriado.

      // Apenas loga o aviso — NÃO lança exceção

Conclusão: O erro de duplicata de ciclo NÃO mais impede a criação do evento.
  O relatório descreve um estado anterior ao código atual.

SEÇÃO 11 — "Erro: Data truncated for column 'action' at row 1"
────────────────────────────────────────────────────────────────────────────────

STATUS: 🔴 PROBLEMA REAL CONFIRMADO — Precisa de migração SQL no VM2.

Análise do código:
  O schema.ts define a coluna action como mysqlEnum com os valores:
    'LOGIN','LOGOUT','VIEW_STUDY','OPEN_VIEWER','CREATE_REPORT','UPDATE_REPORT',
    'SIGN_REPORT','DELETE_REPORT','REVISE_REPORT','CREATE_USER','UPDATE_USER',
    'DELETE_USER','ACTIVATE_USER','DEACTIVATE_USER','CREATE_UNIT','UPDATE_UNIT',
    'DELETE_UNIT','PACS_QUERY','PACS_DOWNLOAD','CREATE_ANAMNESIS',
    'EDIT_STUDY_METADATA','RESET_DOCTOR_BILLING','CREATE_LAYOUT',
    'UPDATE_LAYOUT','DELETE_LAYOUT','BILLING_EVENT_FAILED'

  A última migração que modificou esse enum foi 0033_add_layout_audit_actions.sql
  que NÃO incluiu 'BILLING_EVENT_FAILED'.

  Não existe nenhuma migração SQL que adicionou 'BILLING_EVENT_FAILED' ao enum
  do banco de dados. O schema.ts foi atualizado no código, mas a migração SQL
  correspondente NUNCA FOI GERADA E APLICADA.

  Resultado: quando o código tenta gravar action = 'BILLING_EVENT_FAILED' no
  banco, o MySQL rejeita com "Data truncated for column 'action'" porque esse
  valor não existe no enum do banco.

  IMPACTO REAL: O erro no audit_log é capturado em try/catch separado
  (reports.ts, linha 257: catch (_) { /* audit também não pode bloquear */ })
  então ele NÃO impede a assinatura do laudo. Mas gera ruído nos logs.

Ação necessária (SQL para executar no VM2):
  ALTER TABLE `audit_log`
  MODIFY COLUMN `action` ENUM(
    'LOGIN','LOGOUT','VIEW_STUDY','OPEN_VIEWER',
    'CREATE_REPORT','UPDATE_REPORT','SIGN_REPORT','DELETE_REPORT','REVISE_REPORT',
    'CREATE_USER','UPDATE_USER','DELETE_USER','ACTIVATE_USER','DEACTIVATE_USER',
    'CREATE_UNIT','UPDATE_UNIT','DELETE_UNIT',
    'PACS_QUERY','PACS_DOWNLOAD',
    'CREATE_ANAMNESIS','EDIT_STUDY_METADATA',
    'RESET_DOCTOR_BILLING',
    'CREATE_LAYOUT','UPDATE_LAYOUT','DELETE_LAYOUT',
    'BILLING_EVENT_FAILED'
  ) NOT NULL;

  Esta migração também precisa ser gerada no código para o banco local.

Análise:

  assinatura e, se o ciclo falha, o billing_visit_event não nasce."

    é criado com doctor_cycle_id = NULL e system_cycle_id = NULL
  - O INSERT de billing_visit_events acontece SEMPRE, independente do ciclo
  - O ciclo é apenas filtro de visualização, não pré-requisito

  O princípio "CADA LAUDO FINALIZADO = 1 EVENTO FINANCEIRO" está implementado.
